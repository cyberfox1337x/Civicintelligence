# D1 Rules

## Scope

Applies to:
- D1 query code
- repository or data-access helpers
- Worker handlers that call `env.DB`
- SQL used for reads, writes, filtering, sorting, or pagination
- migrations and schema-aware query changes

## Best Practices

- Use parameterized queries for all dynamic values.
- Prefer `env.DB.prepare(...).bind(...)` over string-built SQL.
- Select only the needed columns.
- Add `LIMIT` to queries that should return bounded results.
- Use explicit `ORDER BY` when result order matters.
- Validate request input before it reaches D1.
- Handle missing rows, empty sets, and failed statements explicitly.
- Keep D1 access patterns easy to trace during review.

## Common Mistakes

- string interpolation in SQL
- `SELECT *` where a narrow projection is enough
- missing `LIMIT` or pagination
- unbounded scans in request paths
- repeated D1 calls inside loops
- trusting client-controlled filters or identifiers without validation
- missing ownership or authorization checks before writes
- leaking raw SQL or database errors in responses
